# H5 小游戏出海新词挖掘工具 V1 需求文档

## 1. 项目定位

项目名称：**Game Keyword Miner**  
中文名：**H5 小游戏出海新词挖掘工具**

V1 定位为：**半自动化 H5 小游戏新词挖掘工作台**。

核心目标：把“新游戏平台发现 + 社媒人工核查 + 趋势验证 + SERP 判断 + 自动评分决策”这套流程产品化，帮助每天稳定筛出最值得上站的 H5 小游戏新词。

---

## 2. 背景逻辑

H5 小游戏新词通常遵循以下传播路径：

```text
游戏源头平台发布
→ 玩家圈层传播
→ YouTube / Reddit / TikTok 等社媒出现讨论
→ 普通用户被动触达
→ 用户主动 Google 搜索
→ Google Trends 开始上涨
→ SEO 工具显示数据
```

所以工具不是找已有搜索量的大词，而是尽早发现：

```text
新发布 + 社媒有热度苗头 + 搜索竞争还低
```

的游戏词。

核心方法：

```text
先找“新”
再判断“热”
最后验证“趋势和竞争”
```

## 3. V1 功能范围

### 3.1 自动完成

1. 自动抓取指定游戏平台的新游戏。
2. 提取游戏名、链接、来源平台、封面、标签等基础信息。
3. 对候选游戏进行清洗和去重。
4. 自动生成候选词记录。
5. 自动生成核查链接：YouTube、Reddit、TikTok、Google Trends、Google SERP。
6. 根据录入指标自动评分。
7. 自动给出决策建议：立即上站、快速做页、加入监测、暂缓。
8. 自动生成 SEO 关键词组合。
9. 自动生成上站页面结构建议。
10. 支持 CSV / Markdown 导出。

### 3.2 半自动完成

| 数据 | V1 方式 |
|---|---|

## 4. 平台采集策略

### 4.1 采集优先级

```text
RSS / Atom
→ 官方或公开 API
→ 普通 HTML 页面解析
→ Playwright 浏览器渲染抓取
→ 手动导入兜底
```

### 4.2 平台支持策略

| 平台 | V1 推荐方式 | 状态 | 说明 |
|---|---|---|---|
| itch.io | RSS 优先，HTML 兜底 | 必做 | 新独立游戏多，HTML5/browser 游戏多 |
| Cocrea | Playwright/HTML 高优先级尝试 | 高优先 | Sprunki 类源头平台，优先检测新作品与新词。采集失败不阻塞其他平台 |
| Scratch | API 优先 | 必做 | 二创和小游戏多，公开数据较多 |
| Game Jolt | API/HTML，Playwright 兜底 | 次优先 | 独立游戏多，但可能动态渲染 |
| CrazyGames | site 搜索验证 | 验证源 | 判断是否被大站收录 |
| Poki | site 搜索验证 | 验证源 | 判断是否被大站收录 |

### 4.3 itch.io 采集

优先入口：

```text
https://itch.io/games/newest
https://itch.io/games/free/newest
https://itch.io/games/html5/newest
https://itch.io/games/platform-web/newest
```

RSS 检测：检查页面源码是否包含 `rss`、`atom`、`feed`、`application/rss+xml`、`application/atom+xml`。如果无 RSS，则使用 HTML 解析。

V1 抓取字段：游戏名、URL、封面、作者、简介、标签、是否 HTML5/browser、发布时间。

### 4.4 Scratch 采集

优先使用公开 JSON 接口或页面数据接口。

入口：

```text
https://scratch.mit.edu/explore/projects/games/
```

过滤规则：

```text
保留：game / simulator / clicker / idle / horror / music / phase / remix 等标题或高互动项目
剔除：Untitled / test / my first game / platformer tutorial / generic game / school project
```

### 4.5 Cocrea 采集

Cocrea 在 Sprunki 类案例中具有较高源头价值（Sprunki 于 2024年8月10日 在 Cocrea 首发，之后才扩散到 YouTube/Reddit/Scratch），V1 将 Cocrea 检测列为高优先级尝试仅次于 itch.io，但**允许采集失败不阻塞整体流程**。

入口：

```text
https://www.cocrea.world/
```

实现目标（按优先级）：

1. **检测 Cocrea 是否存在公开新作品列表、新作品排序、搜索页或站内作品集合页**。
2. **分析页面 Network 请求**，优先寻找公开 JSON/API 数据源。
3. **如果存在可解析 HTML**，则使用 HTML 解析器抓取作品名、作品链接、作者、封面、发布时间或更新时间。
4. **如果页面依赖动态渲染**，则使用 Playwright 作为兜底方案。
5. **如果无法稳定抓取**，则至少提供： Cocrea 快捷入口 + Google site 搜索入口 + 手动导入入口。

> **重要**：文章中 Cocrea 的 Sprunki 首发是通过 Google `site:cocrea.world sprunki` 手动搜索发现的，而非平台主动推送。V1 技术验证后如果 Cocrea 无法稳定采集，**不强制要求阻塞开发**，降级为手动入口即可。

Cocrea 重点抓取字段：

| 字段 | 说明 |
|---|---|
| name | 作品/游戏名称 |
| sourceUrl | Cocrea 作品链接 |
| author | 作者或创作者 |
| thumbnail | 封面图 |
| publishedAt | 平台显示发布时间，若有 |
| updatedAt | 平台显示更新时间，若有 |
| firstSeenOnPlatformAt | 推断首次出现在 Cocrea 的时间 |
| discoveredAt | 工具首次发现时间 |

Cocrea 兜底搜索：

```text
site:cocrea.world "游戏名"
site:cocrea.world "sprunki"
```

要求：**Cocrea 采集失败不能阻塞其他平台同步**，但必须在同步日志中明确记录失败原因、使用的采集方式和下一步建议。

## 5. RSS / Feed 设计

RSS / Atom 用于发现新游戏，不用于判断 YouTube、Reddit、TikTok、Google Trends 或 SERP 热度。

RSS 字段映射：

| RSS 字段 | 候选词字段 |
|---|---|
| title | 游戏名 |
| link | 来源 URL |
| guid | 唯一标识 |
| pubDate | 发布时间 |
| description | 简介 |
| author | 作者 |
| category | 标签 |

RSS 采集流程：

```text
读取平台 feedUrl
→ 请求 RSS/Atom XML
→ 解析 item/entry
→ 提取 title/link/pubDate
→ 标准化游戏名
→ 过滤无效标题
→ 根据 guid/link/name 去重
→ 入库候选词
```

## 6. “新游戏”的定义

V1 中需要区分两个概念：**平台近期新游戏** 和 **工具新发现游戏**。

平台近期新游戏指：**在目标游戏平台上 7 天内第一次出现的新游戏词**。这是业务上最重要的”新词”定义，优先用于判断一个游戏是否具备新词套利价值。（注：Google Trends 数据有3-7天滞后，7天内的窗口可以较好地平衡”够新”和”有数据验证”两个需求）

工具新发现词指：**工具第一次抓取到并入库的游戏词**。工具新发现词不一定等于平台近期新词，例如一个游戏可能三天前已经发布，但工具今天才首次发现。

判断优先级：

1. 平台明确显示的发布时间在 7 天内。
2. RSS / API 返回的 `pubDate`、`createdAt` 或 `publishedAt` 在 7 天内。
3. Sitemap 的 `lastmod` 在 7 天内，但需要注意 `lastmod` 可能只是更新时间，不一定是首次发布时间。
4. 如果平台没有发布时间，则不能强判为平台近期新词，只能标记为工具新发现词。

V1 需要记录以下时间字段：

| 字段 | 含义 |
|---|---|
| publishedAt | 平台显示发布时间 |
| firstSeenOnPlatformAt | 推断的首次出现在平台的时间 |
| updatedAt | 平台显示更新时间 |
| discoveredAt | 工具首次抓到的时间 |
| isPlatformNew7d | 是否为平台 7 天内新词 |
| isFirstDiscoveredByTool | 是否为工具首次发现词 |
| timeConfidence | 时间判断置信度：high / medium / low |
| recencyLevel | 新近度等级：`<24h` / `1-7d` / `7-30d` / `>30d` |

最终判断口径：

```
新游戏词 = 在目标游戏平台上 7 天内第一次出现的新游戏词。
```

如果无法获取平台发布时间，则使用 `discoveredAt` 作为兜底，并在工具中标记为”发布时间未知”或 `timeConfidence = low`。

## 7. 数据清洗与去重

### 7.1 标题标准化

规则：转小写、去多余空格、去版本号、去 demo/prototype/alpha/beta 等版本标签、去括号内非必要信息、去特殊符号。

示例：

| 原始标题 | 标准化名称 |
|---|---|
| Nomad Idle Demo | nomad idle |
| Nomad Idle v0.1 | nomad idle |
| [Prototype] Dark Loop | dark loop |
| Tiny Pasture - Alpha | tiny pasture |

### 7.2 无效标题过滤

剔除：Untitled、Test、My Game、My First Game、Platformer、Demo Game、New Game、Game Jam Project、School Project。

### 7.3 系列词处理

如 `Sprunki Phase 3`、`Sprunki Phase 4`、`Sprunki Retake`，应拆成：

```text
主系列词：Sprunki
具体变体词：Sprunki Phase 4
```

### 7.4 去重规则

| 类型 | 规则 |
|---|---|
| 完全去重 | normalizedName 一样且 sourceUrl 一样 |
| 跨平台去重 | normalizedName 一样但来源不同，合并来源不删除 |
| 疑似重复 | 名称相似，标记为疑似同一游戏，由用户确认 |

## 9. 评分系统

满分 100 分：

| 维度 | 分数 |
|---|---:|
| 新鲜度 | 25 |
| 社媒热度 | 30 |
| 趋势信号 | 20 |
| SERP 竞争 | 15 |
| 可上站性 | 10 |

### 9.1 新鲜度评分

新鲜度评分优先使用平台发布时间或平台首次出现时间，而不是工具发现时间。

判断优先级：

```text
publishedAt
→ RSS/API 的 pubDate / createdAt / publishedAt
→ firstSeenOnPlatformAt
→ discoveredAt，兜底使用
```

如果只能使用 `discoveredAt` 判断，需要降低时间置信度，标记 `timeConfidence = low`。

| 时间 | 分数 | 置信度说明 |
|---|---:|---|
| 平台明确 7 天内首次出现 | 25 | 高 |
| 工具今天首次发现且无平台发布时间 | 20 | 低 |
| 8-30 天 | 12 | 中 |
| 30 天以上 | 3-6 | 低（长尾新词仍可能有传播价值） |

> **注**：Sprunki 等案例表明，从平台首发到社媒爆发可能超过30天。长尾新词不直接给0分，保留进一步观察的可能。

### 9.2 社媒热度评分

社媒热度是判断"热"的核心信号。**有大V/博主视频（>100万播放）是最强的热度信号**，优先于视频数量计算。

| 信号 | 分数 | 说明 |
|---|---:|---|
| 有大V视频（>100万播放） | 12 | YouTube 满分，顶级传播力 |
| 有普通视频（>1万播放） | 8 | 正常热度 |
| 有视频但播放量低 | 4 | 早期信号，继续观察 |
| 无视频 | 0 | 可能还未传播开 |

Reddit 最高 8 分；TikTok 最高 5 分；Discord 讨论 3 分；游戏平台评论最高 2 分。

> **注**：文章案例 Sprunki 的传播路径是 YouTube 大V驱动（1130万观看），表明大V视频是爆款的关键触发点，而非单纯堆视频数量。

### 9.3 趋势信号评分

| 状态 | 分数 |
|---|---:|
| 7 天明显上升 | 20 |
| 14 天明显上升 | 15 |
| 30 天缓慢上升 | 10 |
| 平稳 | 5 |
| 无数据 | 3 |
| 高位平稳且竞争高 | 0 |

### 9.4 SERP 竞争评分

| 竞争状态 | 分数 |
|---|---:|
| 低 | 15 |
| 中 | 10 |
| 高 | 4 |
| 极高 | 0 |

### 9.5 可上站性评分

| 条件 | 分数 |
|---|---:|
| 可 iframe 在线玩 | 10 |
| 可浏览器玩但不能 iframe | 7 |
| 可下载，不能在线玩 | 5 |
| 只能写攻略/资讯 | 3 |
| 风险高或不可访问 | 0 |

### 9.6 决策规则

| 总分 | 决策 |
|---:|---|
| 80-100 | 立即上站 |
| 65-79 | 快速做页 |
| 50-64 | 加入监测 |
| 0-49 | 暂缓 |

字段：candidateId、youtubeVideos7d、topYoutubeViews、topYoutubeComments、hasInfluencerVideo、redditPosts7d、topRedditUpvotes、topRedditComments、tiktokSignal、tiktokTopViews、discordSignal、platformReviews7d、updatedAt。

### 10.4 TrendMetrics 趋势表

字段：candidateId、trend7d、trend30d、mainRegion、notes、updatedAt。

### 10.5 SerpMetrics 竞争表

字段：candidateId、competition、hasOfficialSite、hasDedicatedSites、dominatedByBigSites、notes、updatedAt。

### 10.6 Score 评分表

字段：candidateId、freshnessScore、socialScore、trendScore、competitionScore、launchScore、totalScore、decision、updatedAt。

### 10.7 Platform 平台配置表

字段：id、name、domain、platformType、crawlMethod、feedUrl、apiUrl、listUrl、enabled、priority、lastSyncAt、lastSyncStatus、notes。

## 11. 技术方案

推荐架构：

```text
前端：React + TypeScript + Vite
后端：Node.js + Express 或 Fastify
数据库：SQLite
采集：Node fetch + Cheerio
RSS 解析：rss-parser 或 fast-xml-parser
动态页面：必要时使用 Playwright
任务：手动触发同步，后续可加定时任务
```

架构示意：

```text
React 前端
Dashboard / 候选词 / 详情 / 导出
        ↓ HTTP API

        ↓
RSS 采集器 / API 采集器 / HTML 采集器 / Playwright 采集器
        ↓
SQLite
candidates / metrics / scores / platforms
```


- 可以批量导入候选词。
- 可以查看候选词列表。

- 可以输出决策建议。
- 可以导出 CSV。

- 可以查看 Kongregate 下拉词和 Google 下拉词提示（辅助参考）。

### 13.2 采集验收

|---|---|

## 14. 合规与稳定性要求

1. 只抓公开页面。
2. 不绕过登录。
3. 不抓用户隐私数据。
4. 不高频请求。
5. 给请求加合理延迟。
6. 尊重平台 robots 和服务条款。
7. 对失败平台降级为人工导入。
8. 如果有官方 API，优先使用官方 API。
9. 某个平台失败不影响其他平台同步。
10. 保留同步日志，方便排查。
